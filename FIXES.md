## 13. เปิดไฟล์แล้วเงียบสนิททั้งไฟล์ (playhead ไม่ขยับด้วย)

- **อาการ:** บางเซสชันกดเล่นแล้วไม่มีเสียงเลยกับทุกท่า (เจอกับ story.wav)
- **สาเหตุ:** WebKit ระงับ (suspend) AudioContext ที่ถูกสร้างก่อนผู้ใช้คลิกครั้งแรก
  และ WebAudioPlayer ของ wavesurfer **ไม่เคยเรียก resume() เลย** → เสียงเงียบ
  และนาฬิกา context หยุดเดิน (เลยดูเหมือนแอปค้าง) — อาการเกิดๆ หายๆ ตามจังหวะ
  ว่า context ถูกสร้างก่อนหรือหลัง gesture แรก
- **วิธีแก้:** `ensureAudioRunning()` ใน `src/audio/player.ts` — เช็ค
  `audioContext.state === "suspended"` แล้ว `resume()` ในทุกทางเข้าของการเล่น
  (playPause / playFrom / playRange ล้วนถูกเรียกจาก user gesture อยู่แล้ว)
- **กันเกิดซ้ำ:** เพิ่มทางเข้าการเล่นใหม่เมื่อไหร่ ต้องเรียก ensureAudioRunning ก่อน
- **⚠ รอบ 2 (เสียงหายซ้ำแม้มี resume):** WebAudio ของ WebKit ดับเองได้ทั้งที่สถานะ
  "running" (เช่น หลังสลับอุปกรณ์เสียง) → **เลิกใช้ WebAudio เล่นเสียงถาวร**:
  ถอดรหัสครั้งเดียว → encode เป็น WAV (src/audio/wav.ts) → เล่นผ่าน media element
  ธรรมดาซึ่งเสถียรใน WebKit; ภาพ/เสียง sync กันโดยโครงสร้างเพราะมาจาก buffer
  ถอดรหัสชุดเดียวกัน (คงหลักการข้อ #7) — **กติกาใหม่: playback ต้องกินข้อมูล
  PCM ชุดเดียวกับที่ใช้วาดภาพเสมอ ไม่ว่าจะเล่นผ่านกลไกไหน**

  8000) **ทุกครั้ง** ไม่ล็อกติดตัวที่ตอบก่อน — sidecar ชนะเสมอเมื่อบูตเสร็จ
- **กันเกิดซ้ำ:** ห้ามเปิด dev uvicorn ค้างไว้ตอนทดสอบตัว build; ถ้าเพิ่มพอร์ต
  candidate ใหม่ ต้องเรียงตามลำดับที่ prod ต้องชนะ dev เสมอ

## 15. ปิดแอปแล้ว backend sidecar ยังค้าง (process ผีกิน RAM/พอร์ต)

- **อาการ:** quit แอป (.app build แล้ว) แต่ audioedit-backend ยังรันค้าง —
  เปิดแอปใหม่ชนพอร์ต 8756 หรือกิน RAM สะสม
- **สาเหตุ:** kill child ผูกกับ `WindowEvent::Destroyed` ซึ่งไม่ยิงเมื่อ quit
  ทั้งแอป (Cmd+Q) — process หลักตายก่อน handler ทำงาน
- **วิธีแก้:** ย้ายไปใช้ `app.run(|handle, event| { if RunEvent::Exit {...} })`
  ใน src-tauri/src/lib.rs — Exit ยิงชัวร์ทุกทางออก (quit/close/window) แล้ว
  kill+wait ตัว child
- **กันเกิดซ้ำ:** การจัดการ lifecycle ของ sidecar ต้องผูกกับ RunEvent::Exit
  ไม่ใช่ window event; ทดสอบทุก build: quit แล้ว `pgrep audioedit-backend`
  ต้องเป็น 0

## 16. แถบ "ติดตั้งโมเดล" ค้างทั้งที่โมเดลมีครบ

- **อาการ:** แอป build แล้ว โชว์แถบเหลือง "ยังไม่มีโมเดล AI" ค้าง แม้ปิด-เปิดใหม่
  ทั้งที่ backend (curl ตรงๆ) ตอบ asr:true align:true
- **สาเหตุ:** frontend เช็คโมเดล "ครั้งเดียว" ด้วย flag `modelsChecked` — sidecar
  ตัว frozen ใช้เวลาบูต (import torch) นาน; ถ้าจังหวะเช็คแรกไปโดนช่วงพร่อง แล้ว
  latch ผลค้าง จะไม่เช็คซ้ำอีกเลย แถบเลยค้าง
- **วิธีแก้:** เช็คโมเดลซ้ำทุกรอบ health poll ตราบใดที่ยัง `!modelsReady`
  (หยุดเองเมื่อเจอ; ข้ามถ้ามี download job กำลังรัน) ใน src/main.ts
- **กันเกิดซ้ำ:** อย่า latch สถานะที่ขึ้นกับ sidecar ตอนบูต — poll จนกว่าจะได้
  สถานะที่ถูก; ทดสอบ: เปิดแอปที่มีโมเดลครบ แถบต้องหายเองใน ~10 วิ

## 17. แถบ "ติดตั้งโมเดล" โชว์ตลอด ซ่อนไม่ลง (CSS override attribute hidden)

- **อาการ:** แถบเหลืองค้างแม้โมเดลมีครบ + backend ตอบ true + โค้ดสั่ง
  `modelsBanner.hidden = true` แล้ว; ล้าง cache/rebuild หลายรอบไม่หาย
- **สาเหตุ (ตัวจริง — ไม่ใช่ cache!):** `.models-banner { display: flex }`
  มี specificity เท่ากับ UA rule `[hidden] { display: none }` แต่มาทีหลัง →
  ชนะ → element โชว์ตลอดไม่ว่า attribute hidden จะเป็นอะไร (แถบ .fine-bar
  ก็โดนด้วย). อาการหลอก: ปุ่มทำงาน (modelsReady=true), แต่ CSS ดันโชว์ทับ
- **วิธีแก้:** `[hidden] { display: none !important; }` ที่ต้น styles.css —
  บังคับ hidden ชนะเสมอ (ครอบทุก element ที่ toggle ด้วย .hidden)
- **กันเกิดซ้ำ:** อย่าตั้ง `display:` ให้ element ที่ toggle ด้วย attribute
  `hidden` โดยไม่มี guard; มี `[hidden]{display:none!important}` global แล้ว
  แต่ถ้าเห็น element "ซ่อนไม่ลง" ให้สงสัย CSS override ก่อน cache
